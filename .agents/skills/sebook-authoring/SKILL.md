# SEBook chapter authoring

Static SEBook chapter pages live under `SEBook/**/*.md` with `layout: sebook`, rendered by [`_layouts/sebook.html`](../../../_layouts/sebook.html). They are Markdown + kramdown + Liquid — **not** the in-browser tutorial runtime (those are YAML under `_data/tutorials/`; see [`tutorial-authoring`](../tutorial-authoring/SKILL.md)). This skill captures the conventions that make a chapter render cleanly: how the floating top section-nav is built from your H1s, what the standard page furniture looks like, and which anchor ids you must not break when renaming a heading.

If you only remember four things:

1. **The floating top nav is one entry per `# ` heading.** The layout builds it by splitting rendered `content` on `<h1>` and emitting one `<li>` per kramdown-id'd heading ([`_layouts/sebook.html` lines 32–41](../../../_layouts/sebook.html)). Too many `# `s → a nav that wraps over multiple rows; long `# `s → a row of run-on labels.
2. **Below the first section, `# ` and `## ` render identically.** The render loop emits the *first* H1 as `<h1>` and every subsequent `# ` as `<h2>` ([line 119](../../../_layouts/sebook.html)). Authored `## ` are also `<h2>`. So demoting `# ` → `## ` removes the nav entry **without changing how the page looks** — it is the cheapest knob for a bloated nav.
3. **Renaming a heading changes its anchor id.** Cross-page deep-links of the form `/SEBook/.../slug.html#kramdown-id` exist throughout the book. `grep -rEn "slug\.html#" SEBook/` **before** you rename. If you must rename, pin the old id with `## New Title {#old-id}` or fix every inbound link in the same commit.
4. **Page furniture is `{% include %}`-based and the References section is automatic.** Quiz, flashcards, UML, language-switcher are shared includes / HTML containers — don't reinvent them. `{% cite Key %}` triggers an auto-appended `<section id="references">` ([lines 130–143](../../../_layouts/sebook.html)); don't author your own `# References`.

## The section-nav mechanic (the core insight)

The floating top nav (`.navbar-nav.navbar-right`, [`_layouts/sebook.html` lines 32–41](../../../_layouts/sebook.html)) is generated by walking the rendered `content` and splitting on `<h1`. For every H1 that has an `id="…"`:

```liquid
{% assign heading_text_short = heading_text | split: ':' | first %}
<li><a class="page-scroll" href="#{{ heading_id }}">{{ heading_text_short }}</a></li>
```

Three implications you can lean on:

- **Heading count drives nav rows.** [`SEBook/designprinciples/soc.md`](../../../SEBook/designprinciples/soc.md) authored 20 H1s and the nav wrapped over multiple rows. Aim for ~6–10 H1s on a typical chapter; audit anything above ~12 for demotion candidates.
- **Labels are truncated at the first colon.** `# Solution: Macro Command` shows up as just **"Solution"** in the nav. Parentheses, dashes, and commas are **not** stripped — `# How to Actually Achieve SoC (Mechanisms)` becomes a fat nav button. If you want a long narrative heading, write it as `# Short Label: long elaboration` so the nav gets only the short label.
- **`## ` is invisible to the nav.** Sub-topics belong under `##`. The render loop never emits them as nav entries.

In `<main>`, the first H1 stays an `<h1>`; every later `# ` re-renders as `<h2>` ([line 119](../../../_layouts/sebook.html)). The page is visually flat at h2 below the first section, so promoting / demoting `# ` ↔ `## ` is purely a **nav-presence toggle**, not a typographic change.

## H1 budget and house style

Chapters that read well — Strategy, Command, Singleton — share a small core H1 vocabulary:

`Problem` · `Context` · `Solution` · `Code Example` · `Design Decisions` · `Consequences` · `Related Patterns`

Closing sections vary by chapter: Strategy ends with `Common Examples` and `Practical Guidance: When NOT to Use Strategy`; Command uses `Check Yourself` and `Further Reading`; State opens with `Intent` instead of `Problem`. The core middle stays consistent. Everything more specific — UML diagrams, sub-cases, individual design decisions — is an `##` under the right parent. **Match the core when you can.** If you find yourself promoting "Common Examples" *and* "Good Examples" *and* "Practical Guidance: When NOT to Use Strategy" all to H1, you are inflating the nav — pick one parent and put the others under `##`.

Rules of thumb:

- ~6–10 H1s on a pattern / principle chapter. Above ~12, audit for demotion.
- Keep H1 text short. If it doesn't fit a sticky nav button, shorten it or move the subtitle to the body.
- Prefer `# Label: subtitle` over `# Long narrative label` — colon truncation saves the nav.
- Sub-topics (`## UML Role Diagram`, `## How does X access Y?`) live under their parent H1, not as siblings.
- The `## Flashcards` / `## Quiz` tail belongs under a single trailing `# Practice` (or directly after `# Consequences`), not as its own H1s.

## Multiple `<h1>` is fine

The page already gets an `<h1>` from front matter `title:` — [`_includes/header_project.html` line 16](../../../_includes/header_project.html) renders it as `<h1>{{ include.title }}</h1>`. Then the first content `# ` is *also* an `<h1>` (line 119 of the layout). HTML5 permits multiple h1s and screen readers handle this fine. **Don't add a wrapper `# Introduction` heading just because the page already has a title h1** — write the first body section's heading the way it would naturally read (`# Intent`, `# Problem`).

## Heading order — WCAG 2.2 SC 1.3.1

The audit at [`tests/wcag22-complete-audit.spec.js:897–907`](../../../tests/wcag22-complete-audit.spec.js) walks the rendered DOM and fails on:

- **Skipped levels**: `level > prevLevel + 1` — e.g. `h2` directly followed by `h4`.
- **Missing h1**: any page with no h1 at all. (Not a concern in practice because `header_project.html` always emits one.)

What this means for an SEBook page:

| Move | Allowed? |
|------|----------|
| `# ` → `## ` (demote; body unchanged because of the layout's re-render) | ✅ Always safe |
| `## ` → `### ` (going deeper one level) | ✅ |
| `## ` → `#### ` (skips `###`) | ❌ Fails the audit |
| Repeating `## ` siblings | ✅ |
| First content heading is `## ` (no `# ` in body) | ✅ — the title h1 from header_project.html is already present |

When in doubt: keep adjacent levels within one of each other, and never let `####` appear without an `###` ancestor on the way down.

## Anchor stability — check before renaming

Kramdown auto-generates an `id="…"` for every heading by lowercasing and dash-joining the words; the layout's `<section id="…">` wrapper uses that id. **Renaming a heading changes the id**, and inbound deep-links from other SEBook pages 404 silently. Always grep first:

```bash
# Before renaming any heading in SEBook/<path>/<slug>.md:
slug="solid"  # the file stem, not a heading
grep -rEn "/SEBook/[^\"\\)]+/${slug}\.html#" SEBook/ --include="*.md"
# Or, broader, every anchor-bearing reference to that slug:
grep -rEn "${slug}\.html#" SEBook/ --include="*.md"
```

Cross-references use the **built `.html#…` form** (Jekyll renders `.md` → `.html`), so grep `slug.html#`, not `slug.md#`.

**Currently-referenced anchor targets** (audited from the codebase) — be especially careful renaming headings on these pages:

- [`SEBook/designprinciples/solid.md`](../../../SEBook/designprinciples/solid.md) — the SOLID-principle anchors are linked from many pattern pages. `openclosed-principle-ocp` (6 inbound links), `single-responsibility-principle-srp` (3), plus `dependency-inversion-principle`, `interface-segregation-principle`, `liskov-substitution-principle-lsp`, `single-responsibility-principle`.
- [`SEBook/designpatterns/state.md`](../../../SEBook/designpatterns/state.md) — `how-to-represent-a-state-in-which-the-object-is-never-doing-anything-either-at-initialization-time-or-as-a-final-state` (3 inbound; the very long id is what makes this brittle).
- [`SEBook/designpatterns/strategy.md`](../../../SEBook/designpatterns/strategy.md) — `optional-strategy-with-default-behavior`.
- [`SEBook/designpatterns/command.md`](../../../SEBook/designpatterns/command.md) — `macro-command`.
- [`SEBook/designpatterns/designpatterns.md`](../../../SEBook/designpatterns/designpatterns.md) — `design-patterns-and-refactoring`, `communicating-design-intent-and-context-tailoring`.
- [`SEBook/testing.md`](../../../SEBook/testing.md) — `regression-testing`.
- [`SEBook/development_practices/testdoubles.md`](../../../SEBook/development_practices/testdoubles.md) — `mock-object`, `test-spy`, `test-stub`.
- `regex-tutorial.html` — `anchors`, `character-classes`, `meta-characters`, `quantifiers`. *(Not a `layout: sebook` chapter — it's an interactive tutorial — but SEBook chapters deep-link to its anchors, so the same rename-cautions apply. The interactive tutorial itself is `tutorial-authoring`'s domain.)*

If you must rename one of these, either pin the new heading with `## New Title {#old-id}` so the old id keeps working, **or** grep-and-fix every inbound reference in the same commit. Re-run the grep after editing — Kramdown's id derivation rule (lowercase, dash-join, some punctuation dropped) is not always obvious.

## Standard page furniture

Front matter — minimal:

```yaml
---
title: State Design Pattern
layout: sebook
---
```

Page-furniture includes, used in this order at the end of most chapters:

```liquid
## Flashcards

{% include flashcards.html id="state" %}

## Quiz

{% include quiz.html id="state" %}
```

…where the data lives at `_data/flashcards/state.yml` and `_data/quizzes/state.yml`. See [`quiz-format`](../quiz-format/SKILL.md) for the YAML schema.

**Bibliography** — write `{% cite Gamma1995 %}` citations inline anywhere; the layout auto-appends the References section. See the dedicated section below for entry conventions and the gating rule.

**UML diagrams** — embed via the shared container; the bundled renderer ([`_layouts/sebook.html` lines 739–773](../../../_layouts/sebook.html)) picks them up and re-renders on dark-mode toggle:

```html
<div class="uml-class-diagram-container" data-uml-type="class" data-uml-spec='@startuml
class Context {
  - state: State
  + request(): void
}
@enduml'></div>
```

`data-uml-type` is `class` | `sequence` | `state`. See the [`diagrams`](../diagrams/SKILL.md) skill for picking the right type and [`good-diagrams`](../good-diagrams/SKILL.md) for whether the diagram earns its place at all.

**Multi-language code samples** — wrap in `<div class="inline-language-switcher" data-language-switcher data-default-language="java">…</div>` with `[data-language-option]` buttons and `[data-language-panel]` panels. The shared script in the layout ([lines 641–737](../../../_layouts/sebook.html)) wires up keyboard, ARIA, and the `inert` swap for hidden panels.

**Auto-wired by the layout — you do not author any of this:**

- Dark-mode / highlight / read-aloud toggles ([lines 64–103](../../../_layouts/sebook.html))
- TTS control bar ([lines 436–468](../../../_layouts/sebook.html))
- Bookmark toggle ([lines 96–102, 353–394](../../../_layouts/sebook.html), when `site.bookmarks` is on)
- Sidebar TOC and the mobile menu button ([lines 49–58](../../../_layouts/sebook.html))
- "Skip to main content" link
- The H1-driven top nav (the whole point of this skill)

## Citations and `_bibliography/references.bib`

The bibliography lives at [`_bibliography/references.bib`](../../../_bibliography/references.bib) (~3000 lines, ~hundreds of entries) and is rendered by [jekyll-scholar](https://github.com/inukshuk/jekyll-scholar). Citation style is `harvard-cite-them-right`, sorted by author ascending ([`_config.yml` lines 127–138](../../../_config.yml)). Inline cites render as `(Author, Year)`; the auto-appended References section lists full entries linked to `url` / `doi`.

The sibling [`_bibliography/my.bib`](../../../_bibliography/my.bib) holds the site author's own publications and is **not** picked up by SEBook chapters — the scholar config loads only `references.bib`. Add SEBook citations to `references.bib`, not `my.bib`.

**Adding a new citation is two steps:**

1. **Add the entry to [`_bibliography/references.bib`](../../../_bibliography/references.bib)** with a stable cite key. The project uses mixed-case keys — `Gamma1995`, `brown2014MakeItStick`, `Wake2003INVESTinGoodStories`, `cohn2004user`, `Bacchelli2013`. Any of those shapes is fine; pick one that matches neighbouring entries on the page. **Always include `url` or `doi`** — jekyll-scholar's bibliography template ([`_config.yml` line 133](../../../_config.yml)) wraps each rendered reference in a clickable `<a>` *only* when an entry has one. Without it, the reader cannot click through.

   ```bibtex
   @book{Gamma1995,
     author    = {Gamma, Erich and Helm, Richard and Johnson, Ralph and Vlissides, John},
     title     = {Design Patterns: Elements of Reusable Object-Oriented Software},
     year      = {1995},
     publisher = {Addison-Wesley},
     isbn      = {0-201-63361-2},
     url       = {https://www.oreilly.com/library/view/design-patterns-elements/0201633612/}
   }
   ```

2. **Cite it inline** anywhere in the chapter body:

   ```liquid
   See the original formulation {% cite Gamma1995 %}.
   ```

   Multiple citations in one parenthetical take space-separated keys (not commas, not multiple `{% cite %}` tags):

   ```liquid
   {% cite Bacchelli2013 Czerwonka2015 Beller2014 %}
   ```

**The References section is auto-generated and gated on rendered output.** The layout's `{% if content contains 'class="citation"' %}` check ([`_layouts/sebook.html` line 130](../../../_layouts/sebook.html)) appends `<section id="references">` with `{% bibliography --cited %}` only when the rendered HTML contains a `.citation` span. Consequences:

- A page with no `{% cite %}` calls gets no References section — good, no empty heading.
- A page that already embeds `<div class="bibliography">` (rare; the SEBook landing page does it) is *skipped* by the auto-append, so a hand-curated bibliography won't duplicate.
- The auto-section shows **only cited entries** (`--cited`), not the full bib file. Adding an entry to `references.bib` without citing it has no visible effect.

**Style guidance** — because `harvard-cite-them-right` renders inline cites as `(Gamma et al., 1995)`, write surrounding prose to flow into the parenthetical:

- ✅ `the original GoF formulation {% cite Gamma1995 %} predates these refinements`
- ❌ `as Gamma et al. (1995) {% cite Gamma1995 %} say` (duplicates the author-year)
- ❌ `the GoF book Gamma1995` (the raw key is not a name; cite it instead)

Do **not** author your own `# References` heading — the layout adds `<section id="references"><h2>References</h2>…</section>` for you, and a hand-written one duplicates it.

**Troubleshooting cite output:**

| Symptom | Likely cause | Fix |
|---|---|---|
| Broken-looking citation marker, or build warning about a missing key | Key not in `references.bib`, or typo | `grep -n "^@.*{Gamma1995," _bibliography/references.bib` to confirm the entry exists with the exact key |
| No References section appears though you added a cite | The cite is inside a Liquid `{% raw %}…{% endraw %}` block or a code fence (so it never rendered to `.citation`) | Move the `{% cite %}` into rendered prose |
| Reference is rendered as plain text, not a link | Entry has neither `url` nor `doi` | Add one to the bib entry — see [`_config.yml` line 133](../../../_config.yml) |
| Two References sections on the page | A `<div class="bibliography">` is also being embedded somewhere on the page | Pick one — either remove the manual block, or remove the `{% cite %}` calls so the auto-append doesn't fire |

## Prose style: punctuation and quotation marks

**Punctuation goes inside quotation marks only when it is part of the quoted material.** SEBook uses the "logical" (British) convention rather than the American typesetters' convention that always tucks commas and periods inside. The reason matters specifically for a CS textbook: prose is full of quoted code identifiers, error messages, command names, file paths, and API strings — mis-tucked punctuation changes what the reader sees as the literal string and (worse) what they copy-paste.

- ✅ The compiler emits `expected ';'`. *(Period ends the sentence; the error message ends with `;`, not `;.`.)*
- ✅ The class is called "Strategy", and it has one method. *(Comma is part of the sentence joining the two clauses, not part of the class name.)*
- ✅ Run `git status`, then check the branch. *(Comma after the code span — punctuation is part of the surrounding sentence, not the command.)*
- ❌ The class is called "Strategy," and it has one method. *(The comma is not part of the class name "Strategy".)*
- ❌ The command is "ls -la." *(The period is not part of the command; a reader who copies the quoted text would wrongly believe the command ends with a dot.)*
- ✅ As the manual puts it: "Patterns are about communication. They give designers a vocabulary." *(Both periods are part of the quoted passage — actual sentence-ending punctuation in the original.)*

The same rule applies to backtick code spans (`` `Strategy` ``) and inline identifiers — punctuation goes outside unless it is literally part of the identifier (rare). When in doubt, ask: *would a reader who copies the quoted text expect this punctuation to be inside it?* If not, it goes outside.

## No page-local `<style>`

Markdown allows raw HTML, so it is technically possible to paste a `<style>` block into a chapter — **don't.** Per [`CLAUDE.md`](../../../CLAUDE.md)'s "Style ownership" rule, presentation belongs in `css/`. If your chapter needs a new visual treatment (a callout, a comparison-table style), add a class hook in the markdown and the rule in a CSS module. See [`maintainable-code`](../maintainable-code/SKILL.md) for the structure rule and [`light-dark-mode`](../light-dark-mode/SKILL.md) for the dark-mode requirement.

## Layout variant: `sebook-combined`

[`_layouts/sebook-combined.html`](../../../_layouts/sebook-combined.html) powers the **course-aggregator** pages (`SEBook/CS130.md`, `SEBook/CS35L.md`, `SEBook/all.md`). The top nav there is **not** built from H1s — it iterates `site.data[<page.course>_nav].topics` ([`_layouts/sebook-combined.html` lines 14–43](../../../_layouts/sebook-combined.html)). So on a `sebook-combined` page:

- H1 count and length **do not affect nav bloat.**
- To change what appears in the top nav, edit `_data/CS130_nav.yml` or `_data/CS35L_nav.yml`, not the page.
- The H1-budget guidance above still applies for *readability*, but the "demote `#` → `##` to declutter the nav" trick has no effect.

Most chapters use `layout: sebook`. The `sebook-combined` variant is a small set of aggregator pages.

## Maintenance note

Update this SKILL.md whenever the `sebook` (or `sebook-combined`) layout's heading-to-nav logic, page-furniture includes, anchor conventions, or scholar/bibliography configuration change. The nav-from-H1 mechanic is a subtle invariant that future agents will not derive from reading the surrounding prose — the skill must keep matching the layout. When adding a new include that becomes part of the standard page furniture, list it in "Standard page furniture" above. If the `scholar:` block in `_config.yml` switches citation style, bibliography source, or sort order, update the "Citations and `_bibliography/references.bib`" section. The "currently-referenced anchor targets" list should be re-audited (re-run the grep in that section) whenever a SEBook reorganization renames headings on `solid.md`, `state.md`, `strategy.md`, `command.md`, `designpatterns.md`, `testing.md`, `testdoubles.md`, or `regex-tutorial.md`.
